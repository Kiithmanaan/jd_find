# 已确认产品与工程决策

> 文档性质：现状事实源——描述当前已确认的产品与工程口径，口径变化时原地改写本文档；不承载漂移追踪（见 `docs/50-todo.md`）和历史决策过程（见 `docs/11-technical-implementation-decisions.md`）。

本文档沉淀已确认的业务、产品、部署和协作决策。若本文档与其他文档不一致，按 `docs/README.md` 的优先级规则以本文档为准。

## 1. 产品范围

- 第一阶段按单兵猎头顾问设计，同时保留团队和企业扩展空间。
- 第一阶段支持一次性岗位寻访。
- 同一个岗位可以多次发起一次性寻访。
- 每次寻访是独立 `SearchRun`。
- 同一 `JobProfile` 下多次寻访结果需要叠加、跨 SearchRun 去重、重新排序。
- 候选人结果第一阶段只展示，不进入 ATS 流程。
- 不做面试、Offer、客户推进、候选人长期跟进状态。

## 2. 岗位画像与版本

- AI 可以生成岗位画像建议、条件建议、权重建议。
- 支持逼问式澄清访谈：AI 按七组固定话题逐题提问（一次一问、附推荐答案），访谈完成后产出画像草稿字段（JD 文本、硬性条件文本建议、软性条件、排除信号、搜索关键词）；访谈会话与问答记录持久化。
- 访谈草稿只作为草稿版本预填输入，画像确认仍必须由用户动作触发。
- AI 不自动确认岗位画像。
- `JobProfileConfirmed` 必须由用户动作触发。
- 岗位画像需要版本机制。
- 岗位画像支持配置排除信号（`negativeSignals`）：命中即提示风险的简历特征描述（如跳槽频率高、短任期贴金、方向漂移），由用户以自然语言逐条录入，可为空，不作为画像确认门槛。
- 软性条件每条支持可选的验证方式提示（`verificationHint`）：说明看简历中什么信号才算真正满足该条件（命脉技能 + 验证方式）。
- JD 原文、搜索条件、硬性条件、软性条件、排除信号、软性匹配 prompt 等任意影响筛选或评分的字段变更后，用户重新确认会生成新的 `JobProfileVersion`。
- 已确认的 `JobProfileVersion` 不可编辑。
- 用户编辑已确认画像时，实际创建新的草稿版本。
- 草稿确认后生成新的 confirmed version。
- `SearchRun` 启动时绑定当时的 `JobProfileVersion`。
- `MatchAssessment` 记录评估时使用的 `JobProfileVersion`。
- 历史 `SearchRun` 结果保留。
- 历史候选人可以基于新版本手动批量重新评估，但系统不自动重新评估。

## 3. 重新评估

- 第一阶段只支持用户手动按钮触发历史候选人批量重新评估。
- 重新评估范围为当前 `JobProfile` 下所有历史去重后的候选人。
- 重新评估时先按新版本重新硬筛，再对硬筛通过者重新做软性匹配。
- 同一 `JobProfile` + 同一目标 `JobProfileVersion` 同一时间只允许一个重新评估任务运行。
- 已完成后允许用户再次手动触发。
- 同一候选人、同一 `JobProfileVersion` 只展示并保留最新业务评估结果。
- 批量重新评估覆盖旧业务评估结果。
- 重新评估后不保留旧评估明细。
- AI 审计保留所有调用记录。
- 候选人汇总结果默认按最新 confirmed version 展示。
- 第一阶段不支持切换评估版本。
- 历史版本仅展示画像本身，不展示旧评估明细。
- 未按当前版本评估的候选人单独放在另一个 Tab 中。
- `JobProfileVersion`、候选人汇总和覆盖式重新评估的实现顺序为先重构领域模型，再补前端交互。

## 4. SearchRun 规则

- SearchRun 目标数量由用户启动时配置。
- 默认值为 200。
- 最小值为 10。
- 最大值为 500。
- 目标数量含义是插件抓取或提交的原始候选人数量。
- 达到目标原始提交数量，或插件上报完成，任一满足即 SearchRun 可进入完成条件。
- SearchRun Completed 后，第一阶段拒绝插件继续提交候选人。
- 用户可以手动取消 Running 中的 SearchRun。
- 取消后 SearchRun 进入 `Cancelled`。
- 取消后通知插件停止当前批次。
- 取消后拒绝新增候选人。
- 已完成处理的候选人和评估结果保留。
- 尚未进入 AI 评估的候选人不再评估。
- 插件通过轮询 + 每次提交 API 返回状态获知 SearchRun 是否取消或完成。

用户可见状态：

- `Running`
- `Completed`
- `Interrupted`
- `Failed`
- `Cancelled`

内部状态：

- `Created`
- `Acquired`
- `Deduplicated`
- `HardFiltered`
- `Assessed`

## 5. 候选人、去重与附件

- 求职意向、活跃度只展示，不参与排序。
- 第一阶段按同一 `JobProfile` 跨 SearchRun 去重。
- 全局候选人去重后置。
- fingerprint 生成规则采用插件提供优先，主工程兜底生成。
- fingerprint 兜底优先级建议为来源平台 profile id、profile URL、联系方式、姓名 + 公司 + 职位。
- 候选人信息全部保存，不做脱敏。
- 暂不需要删除能力。
- 暂不需要导出能力。
- 需要支持下载简历附件。
- 第一阶段简历附件使用本地文件存储，后续预留对象存储扩展。
- 第一阶段本地附件按 `SearchRun` 分目录组织。
- 第一阶段所有能访问系统的人都可下载简历附件。
- 第一阶段只支持浏览器插件上传附件。
- 前端暂不支持手动上传附件。
- 候选人提交只通过浏览器插件 Plugin Token 通道，Web 端不提供手动添加候选人入口。
- 插件先提交候选人，再单独上传附件并绑定 `candidateId`。
- 第一阶段每个候选人只允许一个简历附件，后续预留多个附件扩展。
- 重复上传候选人附件时覆盖旧附件。
- 单个简历附件大小限制为 20MB。

## 6. Source 与浏览器插件

- 当前主工程不直接实现真实招聘平台抓取。
- 当前主工程只预留 Source Adapter 和 Plugin Ingestion 接口。
- 后续单独开发浏览器插件负责抓取数据。
- 浏览器插件将数据提交给主工程。
- 主工程负责接收、规范化、去重、硬筛、AI 评估、排序、审计。
- 插件数据提交同时支持单条提交和批量提交。
- 主工程以 `batchId`、`sourcePlatform`、`jobProfileId`、`searchRunId` 做归并。
- 插件接入协议第一阶段按完整协议锁定，覆盖插件登录、Token 获取/保存/过期、候选人单条提交、候选人批量提交、附件上传、批次状态上报、SearchRun 状态轮询和错误码动作。
- 插件抓取后提交无需用户再次确认。
- 插件候选人提交接口快速返回已接收。
- 插件单条提交进入待评估队列，主工程短时间聚合后处理。
- 插件提交后台按 `SearchRun` 异步聚合处理。
- 默认聚合窗口为 30 秒。
- 单个 `SearchRun` 缓冲满 20 条候选人时立即触发评估。
- 批量提交可以直接评估整批。
- 插件提交后 AI 评估采用批次或短时间聚合处理，避免调用过碎。
- 插件提交接口第一阶段做简单保护型限流，维度为 API key + SearchRun。
- 插件错误处理策略：
  - `ValidationError`：插件记录并跳过候选人。
  - `SearchRunCompleted` / `SearchRunCancelled` / `SearchRunFailed`：插件停止当前批次。
  - `AuthError`：插件停止并提示重新配置凭证。
  - `RateLimited`：插件降频后重试。
  - `InternalError`：插件有限重试。

风控定位：

- 风控规避策略属于浏览器抓取插件调度层。
- MVP 中主工程只负责接收插件提交的数据。
- MVP 中主工程不接收、不处理插件风险状态。
- 主工程不负责制定抓取节奏、点击路径、停顿策略、账号使用策略。
- 主工程不配置、不决策、不调度抓取频率或拟人化策略。
- 浏览器插件风控相关能力完全属于插件侧。

## 7. SourceLead

- `SourceLead.url` 失效时，标记 `SourceLead.expired = true`。
- 继续展示 `fallbackClues`。
- 主工程不自动重新搜索。
- 后续可以通过事件或接口通知浏览器插件重新抓取。
- 该部分后续可能调整，第一阶段不要做过重设计。

## 8. AI 与软性匹配

- 主工程保持通用 HTTP AI Adapter，不绑定具体 AI 供应商。
- 具体模型、prompt 和供应商切换可以由外部 AI 服务封装。
- 已提供的 `jd-clarify` skill 第一阶段只作为 prompt 和逻辑参考，不作为主工程运行时依赖。
- 主工程内重写岗位画像生成、软性条件生成、匹配评估和审计记录的 agent 编排。
- AI 允许输出推荐结论。
- AI 需要基于岗位画像判断合适点、不合适点、风险点、推荐结论。
- 画像配置了排除信号时，匹配评估必须逐条对照候选人简历；命中的排除信号优先写入风险点（上限仍为 3 条，超出按严重程度截断）。
- 推荐结论采用固定枚举 + 自然语言说明。
- 推荐结论枚举为 `推荐`、`待定`、`不推荐`。
- 合适点、不合适点、风险点每类最多 3 条。
- 匹配分建议保持 0-100。
- AI 必须覆盖请求范围内候选人，不应返回范围外候选人。
- AI 输出契约第一阶段按完整结构化契约落地，包含推荐枚举、推荐说明、匹配分、合适点、不合适点、风险点、trace、prompt version、agent version、输入快照和输出快照。
- 第一阶段不做强成本预算和用户级额度。
- 需要记录 AI 调用次数、失败次数、耗时、provider、model、prompt 或 skill 版本。
- AI 契约已扩展 `MatchAssessment`，支持推荐结论、不合适点、风险点和 trace；“推荐/待定/不推荐”是匹配评估结论，不等同于自动录用或最终招聘决策。

软性条件：

- 软性条件不采用固定维度配置。
- 软性条件来源于用户输入的自然语言要求。
- 系统根据用户输入自动生成用于匹配的提示词。
- 生成后的提示词用于驱动匹配 agent 执行软性条件匹配。
- 软性条件生成 agent 的细节后续在工程实现阶段讨论。
- 需要保存用户原始软性要求文本、生成后的匹配 prompt、prompt version、agent version、软性匹配输出和审计记录。

AI 审计：

- AI 审计跟随 SearchRun 生命周期保存，第一阶段不单独清理。
- AI 审计展示完整输入快照和输出快照。
- AI 审计保存实际 prompt 内容。
- AI 审计保存 prompt version、agent version、provider、model。
- AI 审计记录成功调用耗时。
- AI 审计记录失败调用错误类型和错误信息。
- 失败调用也应尽量保留输入快照和 prompt。

## 9. 硬性条件与规则配置

硬性条件第一阶段维度：

- 全文关键词
- 城市
- 行业
- 学历
- 最低工作年限

硬性条件采用两段式：

- 淘汰规则。
- 合格规则。
- 硬筛采用规则组 + 单层条件表达。
- 先执行淘汰规则，命中任意淘汰规则直接淘汰。
- 未命中淘汰规则后执行合格规则。
- 命中任意合格规则则硬筛通过。
- 没命中淘汰规则，但也没命中任何合格规则，则淘汰。

规则编辑器：

- 支持多条淘汰规则。
- 支持多条合格规则。
- 多条淘汰规则之间是或关系。
- 多条合格规则之间是或关系。
- 单条规则第一阶段只支持单层条件。
- 单条规则内部选择 `AND` 或 `OR`。
- 第一阶段不支持复杂条件树。

配置：

- 需要有配置表存储硬性条件维度和匹配项。
- 不把硬性维度写死在代码中。
- 硬性条件配置第一阶段通过数据库 seed 维护。
- 城市配置第一批 seed 只覆盖 demo 和近期岗位需要的城市及别名。
- 行业配置第一批 seed 只覆盖 demo 和近期岗位涉及行业及别名。
- 第一阶段不追求城市和行业配置全量覆盖。
- 前端只读取配置。
- 硬性条件配置为全局配置，所有用户共用。
- 条件值来源同时支持配置选项和用户手动输入。

匹配规则：

- 全文关键词支持精确包含和归一化包含。
- 归一化包含忽略大小写、空格、常见标点、全角半角差异。
- 硬性关键词不做语义匹配，语义匹配归软性条件。
- 全文关键词条件支持多个关键词。
- 多关键词支持任一命中和全部命中。
- 城市、行业、学历等配置项需要支持别名。
- 学历按等级比较。
- 最低工作年限按候选人年限大于等于要求年限判断。
- 年限未知第一阶段视为不满足。
- 行业支持标准值/别名 + 多行业任一命中，不做行业层级。
- 城市支持标准值/别名 + 多城市任一命中，支持不限城市，不做城市群、期望城市、远程办公规则。

Trace：

- 硬性条件命中记录需要保存并展示。
- 软性匹配结果需要 trace。

## 11. 认证与权限

- 第一阶段前端需要正式账号登录。
- 登录方式为邮箱 + 密码。
- 用户通过 CLI/脚本创建，不开放注册。
- 登录态采用 JWT。
- JWT 过期时间为 7 天。
- 后续再支持 refresh token。
- 插件也使用邮箱 + 密码登录。
- 插件使用单独 Plugin Token，权限范围比 Web JWT 更窄。
- Web 用户可以看到所有 `JobProfile`。
- Web 用户可以看到所有 `SearchRun`。
- Web 用户可以下载所有候选人简历附件。
- Web 第一阶段不做数据隔离。
- 插件只能向自己的 `SearchRun` 提交数据。
- 插件只能上传自己提交候选人的附件。
- 插件只能上报自己 SearchRun 的批次状态。
- SearchRun owner 为创建 SearchRun 的 Web 登录用户。
- JobProfile 记录创建者，但第一阶段 Web 不按创建者过滤。

## 12. 部署与协作

- 第一阶段部署目标为单台云服务器 Docker Compose。
- PostgreSQL 和 Redis 都使用 Docker Compose 自托管。
- PostgreSQL 必须使用持久化 volume。
- Redis 建议使用 volume 或开启持久化配置。
- 至少需要每日 `pg_dump` 级别数据库备份。
- 第一阶段需要域名和 HTTPS。
- 使用 Caddy 反向代理。
- HTTPS 证书使用 Let's Encrypt。
- 每日 `pg_dump` 备份第一阶段保留 14 天。
- 第一阶段不使用 Docker 镜像仓库，在服务器上 `git pull` 后本机 build。
- Git 分支策略为 `main` 稳定 + feature branch 开发。
- feature branch 命名规范为 `codex/中文短描述`。
- 提交规范采用中文 + 类型前缀，例如 `feat: 增加插件提交接口`。
- 第一阶段暂不需要 PR review。
- 需要 GitHub Actions CI，每次 push 跑后端 typecheck、前端 typecheck、Prisma validate、后端 test、前端 build。
- 合并到 `main` 前本地必须执行全量检查：`npm run typecheck`、`npm run web:typecheck`、`env DATABASE_URL=postgresql://postgres:postgres@localhost:5432/jd_search npm run prisma:validate`、`npm test`、`npm run web:build`。
- 第一阶段需要 OpenAPI YAML/JSON，暂不需要 Swagger UI。
- README 保持简洁，详细中文手册放到 docs。
- 第一阶段需要 demo 脚本 + 验收清单。
- 代码类 TODO 执行前准备最小 demo 岗位画像和候选人数据集，用于驱动接口、筛选、AI 评估和验收。
- 所有修改必须遵循文档驱动实施路径：先确认需求文档，再确认技术设计文档，再确认 TODO 和验收文档，最后修改代码。
- 如修改影响产品口径、架构设计、接口契约、验收标准或待办状态，必须先同步对应文档，再进入代码实现。
- 任何进入实现的 TODO 必须能映射到验收清单中的验收项；验收清单中后置的事项必须在 TODO 文档中保留追踪项。

| 寻访任务 | `SearchRun` / `SearchEvent` | 一次性寻访的运行边界；状态流转由领域状态机控制，事件用于审计与恢复 |
| 候选人结果 | `CandidateResult`（不叫 Candidate） | 属于某画像/寻访任务的候选人业务对象，不升级为长期人才对象 |
| 候选人草稿 | `CandidateDraft` | 插件或来源适配器提交的原始候选人数据，规范化前的形态 |
| 匹配评估 | `MatchAssessment`（不叫 AIAssessment） | 推荐结论（推荐/待定/不推荐）、匹配分、合适点/不合适点/风险点、trace；覆盖式重评估口径见第 3 节 |
| AI 审计记录 | `AIAssessmentAuditRecord` | 每次 AI 调用的输入/输出快照、prompt version、agent version、provider、model、耗时与失败信息；保留所有调用 |
| 原始来源回链 | `OriginalSourceLink`（别名 `SourceLead`） | 回到来源平台的线索对象，含标准化 URL、失效状态和 `fallbackClues`，不得退化为普通 url 字符串 |
| 简历附件 | `ResumeAttachment` | 每个候选人一个附件，插件上传、重复上传覆盖；API 不暴露本地 `storagePath` |
| 硬筛规则 | `HardConditionRule` / `HardConditionDimension` | 两段式：先淘汰规则后合格规则，维度与选项由配置驱动 |
| 指纹 | fingerprint | 候选人去重标识；插件提供优先，主工程按第 5 节优先级兜底生成 |
| 状态枚举 | `SearchRunStatus` / `JobProfileStatus` / `CandidateResultStatus` | 枚举值以 `src/domain/types.ts` 为准，展示语义映射见 `docs/10` 第 2 节 |
