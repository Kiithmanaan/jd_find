# 已确认产品与工程决策

本文档沉淀当前对话中已经确认的业务、产品、部署和协作决策。若本文档与早期架构文档不一致，以本文档为后续实现输入。

## 1. 产品范围

- 第一阶段按单兵猎头顾问设计，同时保留团队和企业扩展空间。
- 第一阶段支持一次性岗位寻访。
- 同一个岗位可以多次发起一次性寻访。
- 每次寻访是独立 `SearchRun`。
- 同一 `JobProfile` 下多次寻访结果需要叠加、跨 SearchRun 去重、重新排序。
- 候选人结果第一阶段只展示，不进入 ATS 流程。
- 不做面试、Offer、客户推进、候选人长期跟进状态。

## 2. 岗位画像与版本

- AI 可以生成岗位画像建议、条件建议、权重建议。
- AI 不自动确认岗位画像。
- `JobProfileConfirmed` 必须由用户动作触发。
- 岗位画像需要版本机制。
- JD 原文、搜索条件、硬性条件、软性条件、软性匹配 prompt 等任意影响筛选或评分的字段变更后，用户重新确认会生成新的 `JobProfileVersion`。
- 已确认的 `JobProfileVersion` 不可编辑。
- 用户编辑已确认画像时，实际创建新的草稿版本。
- 草稿确认后生成新的 confirmed version。
- `SearchRun` 启动时绑定当时的 `JobProfileVersion`。
- `MatchAssessment` 记录评估时使用的 `JobProfileVersion`。
- 历史 `SearchRun` 结果保留。
- 历史候选人可以基于新版本手动批量重新评估，但系统不自动重新评估。

## 3. 重新评估

- 第一阶段只支持用户手动按钮触发历史候选人批量重新评估。
- 重新评估范围为当前 `JobProfile` 下所有历史去重后的候选人。
- 重新评估时先按新版本重新硬筛，再对硬筛通过者重新做软性匹配。
- 同一 `JobProfile` + 同一目标 `JobProfileVersion` 同一时间只允许一个重新评估任务运行。
- 已完成后允许用户再次手动触发。
- 同一候选人、同一 `JobProfileVersion` 只展示并保留最新业务评估结果。
- AI 审计保留所有调用记录。
- 候选人汇总结果默认按最新 confirmed version 展示。
- 第一阶段不支持切换评估版本。
- 历史版本仅展示。
- 未按当前版本评估的候选人单独放在另一个 Tab 中。

## 4. SearchRun 规则

- SearchRun 目标数量由用户启动时配置。
- 默认值为 200。
- 最小值为 10。
- 最大值为 500。
- 目标数量含义是插件抓取或提交的原始候选人数量。
- 达到目标原始提交数量，或插件上报完成，任一满足即 SearchRun 可进入完成条件。
- SearchRun Completed 后，第一阶段拒绝插件继续提交候选人。
- 用户可以手动取消 Running 中的 SearchRun。
- 取消后 SearchRun 进入 `Cancelled`。
- 取消后通知插件停止当前批次。
- 取消后拒绝新增候选人。
- 已完成处理的候选人和评估结果保留。
- 尚未进入 AI 评估的候选人不再评估。
- 插件通过轮询 + 每次提交 API 返回状态获知 SearchRun 是否取消或完成。

用户可见状态：

- `Running`
- `Completed`
- `Interrupted`
- `Failed`
- `Cancelled`

内部状态：

- `Created`
- `Acquired`
- `Deduplicated`
- `HardFiltered`
- `Assessed`

## 5. 候选人、去重与附件

- 求职意向、活跃度只展示，不参与排序。
- 第一阶段按同一 `JobProfile` 跨 SearchRun 去重。
- 全局候选人去重后置。
- fingerprint 生成规则采用插件提供优先，主工程兜底生成。
- fingerprint 兜底优先级建议为来源平台 profile id、profile URL、联系方式、姓名 + 公司 + 职位。
- 候选人信息全部保存，不做脱敏。
- 暂不需要删除能力。
- 暂不需要导出能力。
- 需要支持下载简历附件。
- 第一阶段简历附件使用本地文件存储，后续预留对象存储扩展。
- 第一阶段所有能访问系统的人都可下载简历附件。
- 第一阶段只支持浏览器插件上传附件。
- 前端暂不支持手动上传附件。
- 插件先提交候选人，再单独上传附件并绑定 `candidateId`。
- 第一阶段每个候选人只允许一个简历附件，后续预留多个附件扩展。
- 重复上传候选人附件时覆盖旧附件。

## 6. Source 与浏览器插件

- 当前主工程不直接实现真实招聘平台抓取。
- 当前主工程只预留 Source Adapter 和 Plugin Ingestion 接口。
- 后续单独开发浏览器插件负责抓取数据。
- 浏览器插件将数据提交给主工程。
- 主工程负责接收、规范化、去重、硬筛、AI 评估、排序、审计。
- 插件数据提交同时支持单条提交和批量提交。
- 主工程以 `batchId`、`sourcePlatform`、`jobProfileId`、`searchRunId` 做归并。
- 插件抓取后提交无需用户再次确认，主工程立即进入处理流程。
- 插件单条提交进入待评估队列，主工程短时间聚合后处理。
- 批量提交可以直接评估整批。
- 插件提交后 AI 评估采用批次或短时间聚合处理，避免调用过碎。
- 插件提交接口第一阶段做简单保护型限流，维度为 API key + SearchRun。
- 插件错误处理策略：
  - `ValidationError`：插件记录并跳过候选人。
  - `SearchRunCompleted` / `SearchRunCancelled` / `SearchRunFailed`：插件停止当前批次。
  - `AuthError`：插件停止并提示重新配置凭证。
  - `RateLimited`：插件降频后重试。
  - `InternalError`：插件有限重试。

风控定位：

- 风控规避策略属于浏览器抓取插件调度层。
- 主工程不负责制定抓取节奏、点击路径、停顿策略、账号使用策略。
- 主工程可以接收插件上报的风险状态。
- 插件应负责尽可能降低平台风控风险。

## 7. SourceLead

- `SourceLead.url` 失效时，标记 `SourceLead.expired = true`。
- 继续展示 `fallbackClues`。
- 主工程不自动重新搜索。
- 后续可以通过事件或接口通知浏览器插件重新抓取。
- 该部分后续可能调整，第一阶段不要做过重设计。

## 8. AI 与软性匹配

- 主工程保持通用 HTTP AI Adapter，不绑定具体 AI 供应商。
- 具体模型、prompt 和供应商切换可以由外部 AI 服务封装。
- AI 允许输出推荐结论。
- AI 需要基于岗位画像判断合适点、不合适点、风险点、推荐结论。
- 匹配分建议保持 0-100。
- AI 必须覆盖请求范围内候选人，不应返回范围外候选人。
- 生成岗位画像和评级简历的具体逻辑后续参考用户提供的已训练 skill。
- 第一阶段不做强成本预算和用户级额度。
- 需要记录 AI 调用次数、失败次数、耗时、provider、model、prompt 或 skill 版本。

软性条件：

- 软性条件不采用固定维度配置。
- 软性条件来源于用户输入的自然语言要求。
- 系统根据用户输入自动生成用于匹配的提示词。
- 生成后的提示词用于驱动匹配 agent 执行软性条件匹配。
- 软性条件生成 agent 的细节后续在工程实现阶段讨论。
- 需要保存用户原始软性要求文本、生成后的匹配 prompt、prompt version、agent version、软性匹配输出和审计记录。

AI 审计：

- AI 审计跟随 SearchRun 生命周期保存，第一阶段不单独清理。
- AI 审计展示完整输入快照和输出快照。
- AI 审计保存实际 prompt 内容。
- AI 审计保存 prompt version、agent version、provider、model。
- AI 审计记录成功调用耗时。
- AI 审计记录失败调用错误类型和错误信息。
- 失败调用也应尽量保留输入快照和 prompt。

## 9. 硬性条件与规则配置

硬性条件第一阶段维度：

- 全文关键词
- 城市
- 行业
- 学历
- 最低工作年限

硬性条件采用两段式：

- 淘汰规则。
- 合格规则。
- 先执行淘汰规则，命中任意淘汰规则直接淘汰。
- 未命中淘汰规则后执行合格规则。
- 命中任意合格规则则硬筛通过。
- 没命中淘汰规则，但也没命中任何合格规则，则淘汰。

规则编辑器：

- 支持多条淘汰规则。
- 支持多条合格规则。
- 多条淘汰规则之间是或关系。
- 多条合格规则之间是或关系。
- 单条规则第一阶段只支持单层条件。
- 单条规则内部选择 `AND` 或 `OR`。
- 第一阶段不支持复杂条件树。

配置：

- 需要有配置表存储硬性条件维度和匹配项。
- 不把硬性维度写死在代码中。
- 硬性条件配置第一阶段通过数据库 seed 维护。
- 前端只读取配置。
- 硬性条件配置为全局配置，所有用户共用。
- 条件值来源同时支持配置选项和用户手动输入。

匹配规则：

- 全文关键词支持精确包含和归一化包含。
- 归一化包含忽略大小写、空格、常见标点、全角半角差异。
- 硬性关键词不做语义匹配，语义匹配归软性条件。
- 全文关键词条件支持多个关键词。
- 多关键词支持任一命中和全部命中。
- 城市、行业、学历等配置项需要支持别名。
- 学历按等级比较。
- 最低工作年限按候选人年限大于等于要求年限判断。
- 年限未知第一阶段视为不满足。
- 行业支持标准值/别名 + 多行业任一命中，不做行业层级。
- 城市支持标准值/别名 + 多城市任一命中，支持不限城市，不做城市群、期望城市、远程办公规则。

Trace：

- 硬性条件命中记录需要保存并展示。
- 软性匹配结果需要 trace。

## 10. 前端

- 第一阶段做极简运维/演示页面。
- 前端技术栈选择 React + Vite。
- 前端构建后由 Nginx/Caddy 托管。
- API 增加 `/api` 前缀。

页面结构：

- 登录页。
- JobProfile 列表页。
- JobProfile 创建/编辑/确认页。
- JobProfile 详情页：
  - 版本历史。
  - 当前版本硬性条件。
  - 当前版本软性条件文本/prompt。
  - 启动 SearchRun。
  - 触发历史候选人重新评估。
  - 候选人汇总 Tab。
- SearchRun 详情页：
  - 状态。
  - 原始提交数量。
  - 候选人处理进度。
  - 候选人列表。
  - 匹配分悬浮详情。
  - AI 审计入口。
- AI 审计详情页或弹窗。

候选人列表直接展示：

- 姓名。
- 当前职位。
- 城市。
- 学历。
- 工作年限。
- 行业。
- 匹配分。
- 求职意向。
- 活跃度。
- 来源平台。
- SourceLead URL。
- fallbackClues。
- 简历附件下载入口。

匹配详情悬浮弹窗展示：

- 推荐结论。
- 合适点。
- 不合适点。
- 风险点。

## 11. 认证与权限

- 第一阶段前端需要正式账号登录。
- 登录方式为邮箱 + 密码。
- 用户通过 CLI/脚本创建，不开放注册。
- 登录态采用 JWT。
- JWT 过期时间为 7 天。
- 后续再支持 refresh token。
- 插件也使用邮箱 + 密码登录。
- 插件使用单独 Plugin Token，权限范围比 Web JWT 更窄。
- Web 用户可以看到所有 `JobProfile`。
- Web 用户可以看到所有 `SearchRun`。
- Web 用户可以下载所有候选人简历附件。
- Web 第一阶段不做数据隔离。
- 插件只能向自己的 `SearchRun` 提交数据。
- 插件只能上传自己提交候选人的附件。
- 插件只能上报自己 SearchRun 的批次状态。
- SearchRun owner 为创建 SearchRun 的 Web 登录用户。
- JobProfile 记录创建者，但第一阶段 Web 不按创建者过滤。

## 12. 部署与协作

- 第一阶段部署目标为单台云服务器 Docker Compose。
- PostgreSQL 和 Redis 都使用 Docker Compose 自托管。
- PostgreSQL 必须使用持久化 volume。
- Redis 建议使用 volume 或开启持久化配置。
- 至少需要每日 `pg_dump` 级别数据库备份。
- 第一阶段需要域名和 HTTPS。
- 使用 Nginx 或 Caddy 反向代理。
- HTTPS 证书使用 Let's Encrypt。
- 第一阶段不使用 Docker 镜像仓库，在服务器上 `git pull` 后本机 build。
- Git 分支策略为 `main` 稳定 + feature branch 开发。
- 提交规范采用中文 + 类型前缀，例如 `feat: 增加插件提交接口`。
- 第一阶段暂不需要 PR review。
- 需要 GitHub Actions CI，每次 push 跑 typecheck/test/Prisma validate。
- 第一阶段需要 OpenAPI YAML/JSON，暂不需要 Swagger UI。
- README 保持简洁，详细中文手册放到 docs。
- 第一阶段需要 demo 脚本 + 验收清单。

## 13. 当前代码需对齐的差异

以下是当前代码骨架与最新确认决策之间的主要差异，后续实现需要调整：

- 当前 API 尚未加 `/api` 前缀。
- 当前仍是批处理 Mock/CSV SearchRun 模型，尚未支持插件增量 ingestion。
- 当前单次目标数量固定为 200，尚未支持用户配置 10-500。
- 当前去重范围主要在单次 SearchRun 内，尚未实现同一 JobProfile 跨 SearchRun 去重。
- 当前 `JobProfile` 尚无版本模型。
- 当前 `MatchAssessment` 尚未包含推荐结论、不合适点、完整 trace、prompt/agent version。
- 当前 AI 契约仍禁止最终推荐措辞，需要按最新决策移除。
- 当前 AI 审计尚未记录 prompt、duration、失败调用信息。
- 当前硬性条件还是固定领域结构，尚未改为配置表 + 淘汰/合格规则组。
- 当前尚无用户、登录、JWT、Plugin Token。
- 当前尚无附件模型和本地文件存储。
- 当前尚无 React/Vite 前端。
- 当前尚无 CI、OpenAPI、demo 脚本和验收清单。
