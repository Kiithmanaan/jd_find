# 文档索引与实施路径

> 文档性质：规范——定义文档驱动框架、阅读顺序、冲突处理和落地路径，是文档驱动流程的唯一事实源。`AGENTS.md` 的流程章节引用本文档，不另行维护副本。

本文档是项目实施前的文档入口。任何功能、修复、重构、接口、数据模型、前端交互、AI 能力、插件协议、部署或运维相关修改，都必须先按本文档确认阅读顺序和联动关系。

## 1. 文档驱动框架

框架由两条正交的轴组成：**主题轴**回答"这个内容归哪份文档"，**性质轴**回答"这份文档按什么纪律维护"。每份文档属于一个主题、具有一种性质，并在头部用一行声明自己的性质。

### 1.1 性质轴：四种维护纪律

| 性质 | 回答的问题 | 更新纪律 |
|---|---|---|
| 规范 | 怎么工作 | 低频、原地改；单一事实源，其他文档只引用不复制 |
| 现状事实源 | 当前口径与设计是什么 | 原地改写，永远反映当前；不承载历史和过程 |
| 决策记录 | 为什么当时这么定 | append-only；过时项标注"已被取代"，不改写、不删除；不参与冲突裁决 |
| 追踪 | 进行到哪了 | 高频更新；前瞻（待办）、现状（验收）、回顾（变更记录）三者分开，单向流动 |

### 1.2 主题轴：十二类职责与本项目映射

编号约定：`0x` 需求、`1x` 架构与决策、`2x` 模块专项需求、`3x` 接口契约、`4x` 开发规约、`5x` 追踪、`6x` 运维、`7x` 演示。

| # | 主题 | 职责 | 性质 | 本项目文档 |
|---|---|---|---|---|
| 1 | 索引/元文档 | 文档体系结构、阅读顺序、冲突裁决、联动流程 | 规范 | `docs/README.md`（本文档） |
| 2 | 需求建模与基线 | 产品范围、业务规则、已确认口径、冲突裁决源、领域词汇表 | 现状事实源 | `00-requirements-baseline.md` |
| 3 | 总体技术架构 | 业务模型到工程边界的转译、分层与依赖方向、状态机全景、数据流、跨模块边界 | 现状事实源 | `10-technical-architecture.md` |
| 4 | 决策记录 | 技术选型、边界决策、否决决策（"决定不做什么"）及其理由 | 决策记录 | `11-technical-implementation-decisions.md` |
| 5 | 模块专项需求 | 单个模块的详细需求与业务规则；细化基线，不复述基线 | 现状事实源 | `20-requirements-frontend-product.md`、`21-requirements-ai-agent-contract.md`、`22-requirements-hard-filter-config.md` |
| 6 | 接口契约 | API、跨端协议、系统配置等对仓库外消费者的承诺；需版本化，尽量机器校验 | 现状事实源 | `30-technical-plugin-protocol.md`、`31-technical-openapi.yaml`、`.env.example`（系统配置，仓库根目录） |
| 7 | 开发规约 | 本地开发、分支提交、架构边界约束、验证命令，以及文档流程的执行要求 | 规范 | `40-engineering-development-guide.md`；`AGENTS.md` 为 agent 侧入口，引用本体系 |

| 11 | 部署/运维 | 部署形态、环境要求、备份恢复、排障 | 现状事实源 | `60-operations-deployment-guide.md`、`61-operations-runbook.md` |

**内部实现设计不设持久文档。** 架构层以下的模块内部设计，以代码、类型和测试为事实源，不写散文文档。判断一份内容是否值得进入文档体系的标准是"谁在代码之外需要这个承诺"：有仓库外消费者的进接口契约；要留给未来决策参考的进决策记录；用于验收的进验收项；都不是，则不写。

### 1.3 横切规则

- **一处事实，多处引用**：任何规则、枚举、清单只在一份文档里以正文形式存在，其他位置用链接指向。
- **能机器校验的不靠人工纪律**：契约一致性用 `npm run contracts:check` 校验，覆盖范围应随契约增长而扩展。
- **文档头部自报性质**：新建文档时必须声明主题与性质；性质决定它被读到时的解释方式——现状文档过时了该改它，决策记录"过时"是常态、只标注不改写。
- **追踪层单向流动**：待办 → 验收 → 变更记录，方向不可逆，完成叙事不得滞留在待办中（细则见第 5 节）。

## 2. 修改前必读顺序

1. `docs/00-requirements-baseline.md`：最高优先级需求与工程决策基线。
2. `docs/10-technical-architecture.md`：总体技术架构。
3. 相关模块专项需求或接口契约文档：前端、AI、硬筛、插件、OpenAPI 等。
4. `docs/50-todo.md`：确认待办、后置事项和风险追踪。
5. `docs/51-acceptance-checklist.md`：确认验收项、验证方式、通过标准和当前状态。
6. 现有代码：在文档约束下阅读实现，不得跳过文档直接改代码。

## 3. 冲突处理规则

当文档之间存在冲突时，按以下优先级处理：

1. `00-requirements-baseline.md`
2. 当前模块专项需求或接口契约文档。
3. `10-technical-architecture.md`
4. 其他专项技术设计。
5. `50-todo.md`
6. `51-acceptance-checklist.md`
7. 演示、部署、运维类文档。
8. `README.md`

若高优先级文档与低优先级文档冲突，必须以高优先级文档为准，并同步修正低优先级文档。决策记录（`11`）不参与裁决：其条目与现状事实源冲突时视为已被取代，在存档中标注而非改写。

## 4. TODO、验收与变更记录联动

- TODO 记录“要做什么、为什么后置、阻塞条件是什么”，只保留未完成事项。
- 验收文档记录“做到什么程度算完成、怎么验证、当前状态是什么”。
- CHANGELOG 记录“已经完成了什么、如何验证的”。
- TODO 中进入实现的事项，必须在验收文档中有对应验收项。
- 验收文档中标记为后置的事项，必须在 TODO 中有对应追踪项。
- 事项完成后单向流动：验证记录写入 `CHANGELOG.md` → 更新验收状态 → 从 TODO 中移除。完成叙事不得滞留在 TODO 中。
- 新需求若暂不实现，必须进入 TODO；若决定实现，必须进入验收文档。

## 5. 标准落地路径

涉及文档修改时，必须按以下顺序执行：

1. 更新需求文档（基线或模块专项需求）。
2. 更新技术设计文档（架构或接口契约）；涉及"为什么这么定"的取舍时，在决策记录中补一条。
3. 更新 TODO 文档。
4. 更新验收文档。
5. 修改代码。
6. 运行验收文档中对应的验证命令或测试场景（涉及契约时运行 `npm run contracts:check`）。
7. 完成后按第 4 节联动规则更新 CHANGELOG、验收状态和 TODO。
8. 汇报读取了哪些文档、修改了哪些文档、验收项如何更新、运行了哪些验证。
